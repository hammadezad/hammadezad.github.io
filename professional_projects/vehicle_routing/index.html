Vehicle Routing Optimizer

Production routing for a national bottling company using a three-phase Gurobi pipeline with warm starts and flexible quantities.

This project plans daily beverage deliveries across hundreds of orders with strict business rules: vehicle capacities, rate-card limits (max stops, max route time, max distance between stops), delivery time windows, and per-location “flex” tolerances.

Problem

Business need. Dispatch the right vehicles on cost-efficient routes while respecting:

  • Vehicle capacities and per-rate limits (stops, route time, inter-stop distance)
  • Delivery time windows and service times at each stop
  • Location-specific over/under shipment “flex” bounds

Pain with legacy system.

  • 60+ minutes per run for ~100 orders
  • Rigid modeling; difficult to change business rules
  • Manual report collation; error-prone daily operations

Solution

  • Three-phase optimizer with warm starts: subcluster → cluster → final allocation
  • Scenario-driven configs: all limits & flags loaded from CSV/DB per scenario
  • Robust logging (rotating logs + error file) and auto-generated shipment/order/route reports

How we solved it

Pipeline with warm starts and reusable constraint builders.

run_pipeline()
 ├─ load_data_from_dataframes()
 ├─ build_global_rate_dicts()          # FC/MC/SC, WCap, MaxStops, MaxRouteDuration, MaxDist, VC
 ├─ solve_phase0_subcluster()          # Subcluster models
 ├─ solve_phase1_for_cluster()         # Union of subclusters; warm-start from Phase 0
 ├─ flex_allocation_and_phase2()       # Final adjustments + flex constraints
 └─ write_*_csv()                      # shipments, orders, routes, scenario stats

Warm starts that actually cut runtime:

# Example: Phase-1 warms x, y, z, q from Phase-0 keys if available
var = model.addVar(vtype=GRB.BINARY, name=f"x_{i}_{j}_{k}")
key = (wave_id, cluster_id, "PHASE0", i, j, k)
if key in X_Param:
    var.Start = X_Param[key]

Key constraints enforced (selected):

  • Capacity: ∑m,p q[m,p,k]·weight[p] ≤ ∑r y[k]·VC[k,r]·WCap[r]
  • Stops: ∑ x[i,j,k] ≤ ∑r y[k]·VC[k,r]·MaxStops[r]
  • Route duration & idle time between stops bounded per rate and scenario flags
  • Flex windows per location + global percent guards (if enabled)
  • Time windows: arrival sequencing and service-time aware constraints

Impact

  • Runtime: ~60 min ➜ < 5 min for 100+ orders (repeatable daily)
  • Cost: ≈ 15% logistics savings via better vehicle utilization & flex control
  • Ops: Zero-touch daily runs with clean reports for planners and partners
  • Maintainability: New rules plug in via config (no model rewrites)
